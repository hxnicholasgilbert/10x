# Phase 6: Push PR

Only execute this phase if the user confirmed they want to push a PR.

---

## Step 1 — Quick checks

Output this and wait for the user's response:

---

**Quick checks** — which would you like to run before pushing?

- [ ] Review PR description
- [ ] Review the diff
- [ ] Run tests

Reply with the numbers (e.g. `1 3`), `all`, or just hit enter to skip.

---

Run any selected quick checks:

**1 — Review PR description:** Output the full contents of `proposed-solution.md`.

**2 — Review the diff:**
```bash
git -C /absolute/path/to/repo diff <BASE_BRANCH>...<TICKET_ID> --stat
git -C /absolute/path/to/repo diff <BASE_BRANCH>...<TICKET_ID>
```

**3 — Run tests:** Detect test command from `package.json` (`test`, `test:unit`, `jest`, `vitest`). Run it and report pass/fail. If tests fail, ask whether to proceed or fix first.

---

## Step 2 — Push draft PR

For each repo in the workspace, push the branch and open a **draft** PR:

```bash
git -C /absolute/path/to/repo push origin <TICKET_ID>

gh pr create \
  --draft \
  --title "[<TICKET_ID>] <summary>" \
  --body "$(cat /Users/<username>/xnet/tickets/<TICKET_ID>/proposed-solution.md)" \
  --base <BASE_BRANCH> \
  --head <TICKET_ID>
```

Output:
```
🔖 Draft PR opened for <repo-name>:
   <PR_URL>
```

Store `PR_URL` — used by deep check agents and Phase 7.

Then ask:
```
Open in browser? y/n
```
If yes: `gh pr view --web`

---

## Step 3 — Deep checks

Output this and wait for the user's response:

---

**Deep checks** — agents will post their findings directly as PR comments. Want any?

- [ ] QA agent — verify acceptance criteria against the code
- [ ] Security agent — scan the diff for vulnerabilities
- [ ] Code review agent — bug and pattern review on changed files

Reply with the numbers (e.g. `1 2`), `all`, or just hit enter to skip.

---

Spawn all selected agents **in parallel**. Pass each agent:
- The PR URL
- The absolute repo path
- The diff output
- The ticket acceptance criteria (from `ticket-breakdown.md`)

Each agent posts findings using both mechanisms as appropriate:

**Inline comments** — for line-specific findings (bugs, security issues, explanations of non-obvious code, suggestions tied to a specific location). Use the file path and line number from the diff:
```bash
gh api repos/<owner>/<repo>/pulls/<PR_NUMBER>/comments \
  --method POST \
  --field path="<file>" \
  --field line=<line_number> \
  --field body="<finding>"
```

**General PR comment** — for summary-level findings, overall status, or anything not tied to a specific line:
```bash
gh pr review <PR_URL> --comment --body "<summary>"
```

Format the general comment:
```
## 🔍 QA Review / 🔐 Security Review / 👁 Code Review

<high-level findings — use checkboxes for each AC or issue>

**Status:** ✅ All clear / ⚠️ Issues found
```

Wait for all agents to complete before proceeding.

---

## Step 4 — Mark ready

After all checks complete, output findings summary and ask:

```
All checks done.

Mark PR as ready for review?
  [y] Mark ready — gh pr ready
  [n] Keep as draft — make changes first, then run /xnet review <TICKET_ID> to handle feedback
```

If yes:
```bash
gh pr ready <PR_URL>
```

Output:
```
✅ PR ready for review: <PR_URL>
```

---

## Error Handling

| Situation | Action |
|-----------|--------|
| `gh` not installed | Halt — tell user to install: `brew install gh && gh auth login` |
| `git push` fails (auth) | Ask user to check `gh auth status` and retry |
| `git push` fails (diverged) | Show error — do not force push. Ask user to resolve manually |
| PR already exists for this branch | Output existing PR URL, skip creation — proceed to Step 3 |
| Tests fail | Surface output, ask whether to proceed or fix first |
| Agent fails to post PR comment | Surface findings in chat instead, note the comment failed |
