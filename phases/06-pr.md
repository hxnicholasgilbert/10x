# Phase 6: Push PR

Only execute this phase if the user confirmed they want to push a PR.

---

## Step 1 тАФ Quick checks

Output this and wait for the user's response:

---

**Quick checks** тАФ which would you like to run before pushing?

- [ ] Review PR description
- [ ] Review the diff
- [ ] Run tests

Reply with the numbers (e.g. `1 3`), `all`, or just hit enter to skip.

---

Run any selected quick checks:

**1 тАФ Review PR description:** Output the full contents of `proposed-solution.md`.

**2 тАФ Review the diff:**
```bash
git -C /absolute/path/to/repo diff <BASE_BRANCH>...<TICKET_ID> --stat
git -C /absolute/path/to/repo diff <BASE_BRANCH>...<TICKET_ID>
```

**3 тАФ Run tests:** Detect test command from `package.json` (`test`, `test:unit`, `jest`, `vitest`). Run it and report pass/fail. If tests fail, ask whether to proceed or fix first.

---

## Step 2 тАФ Push draft PR

For each repo in the workspace, push the branch and open a **draft** PR:

```bash
git -C /absolute/path/to/repo push origin <TICKET_ID>

gh pr create \
  --draft \
  --title "[<TICKET_ID>] <summary>" \
  --body "$(cat /Users/<username>/xnet/tickets/<TICKET_ID>/proposed-solution.md)" \
  --base <BASE_BRANCH> \
  --head <TICKET_ID>
```

Output:
```
ЁЯФЦ Draft PR opened for <repo-name>:
   <PR_URL>
```

Store `PR_URL` тАФ used by deep check agents and Phase 7.

Then ask:
```
Open in browser? y/n
```
If yes: `gh pr view --web`

---

## Step 3 тАФ Deep checks

Output this and wait for the user's response:

---

**Deep checks** тАФ agents will post their findings directly as PR comments. Want any?

- [ ] QA agent тАФ verify acceptance criteria against the code
- [ ] Security agent тАФ scan the diff for vulnerabilities
- [ ] Code review agent тАФ bug and pattern review on changed files

Reply with the numbers (e.g. `1 2`), `all`, or just hit enter to skip.

---

Spawn all selected agents **in parallel**. Pass each agent:
- The PR URL
- The absolute repo path
- The diff output
- The ticket acceptance criteria (from `ticket-breakdown.md`)

Each agent must post its findings as a PR review comment:
```bash
gh pr review <PR_URL> --comment --body "<agent findings>"
```

Format each comment clearly:
```
## ЁЯФН QA Review / ЁЯФР Security Review / ЁЯСБ Code Review

<findings тАФ use checkboxes for each AC or issue>

**Status:** тЬЕ All clear / тЪая╕П Issues found
```

Wait for all agents to complete before proceeding.

---

## Step 4 тАФ Mark ready

After all checks complete, output findings summary and ask:

```
All checks done.

Mark PR as ready for review?
  [y] Mark ready тАФ gh pr ready
  [n] Keep as draft тАФ make changes first, then run /xnet review <TICKET_ID> to handle feedback
```

If yes:
```bash
gh pr ready <PR_URL>
```

Output:
```
тЬЕ PR ready for review: <PR_URL>
```

---

## Error Handling

| Situation | Action |
|-----------|--------|
| `gh` not installed | Halt тАФ tell user to install: `brew install gh && gh auth login` |
| `git push` fails (auth) | Ask user to check `gh auth status` and retry |
| `git push` fails (diverged) | Show error тАФ do not force push. Ask user to resolve manually |
| PR already exists for this branch | Output existing PR URL, skip creation тАФ proceed to Step 3 |
| Tests fail | Surface output, ask whether to proceed or fix first |
| Agent fails to post PR comment | Surface findings in chat instead, note the comment failed |
