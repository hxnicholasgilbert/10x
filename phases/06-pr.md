# Phase 6: Push PR

Only execute this phase if the user confirmed they want to push a PR.

## Step 1 — Quick checks

Output this and wait for the user's response:

---

**Quick checks** — which would you like to run before pushing?

- [ ] Review PR description
- [ ] Review the diff
- [ ] Run tests

Reply with the numbers (e.g. `1 3`), `all`, or just hit enter to skip.

---

Parse the response and note which quick checks to run.

## Step 2 — Deep checks

Output this and wait for the user's response:

---

**Deep checks** — these spawn agents and take a few minutes. Want any?

- [ ] QA agent — verify acceptance criteria against the code
- [ ] Security agent — scan the diff for vulnerabilities
- [ ] Code review agent — bug and pattern review on changed files

Reply with the numbers (e.g. `1 2`), `all`, or just hit enter to skip.

---

Parse the response and note which deep checks to run.

---

## Step 2 — Run selected checks

Run quick checks first (inline), then deep checks in parallel.

### Quick checks

**a — Review PR description**

Output the full contents of `proposed-solution.md` for the user to read.

**b — Review the diff**

```bash
git -C /absolute/path/to/repo diff <BASE_BRANCH>...<TICKET_ID> --stat
git -C /absolute/path/to/repo diff <BASE_BRANCH>...<TICKET_ID>
```

Output the result.

**c — Run tests**

Detect the test command from `package.json` scripts (`test`, `test:unit`, `jest`, `vitest`). Run it:

```bash
cd /absolute/path/to/repo && <test-command>
```

Report pass/fail. If tests fail, surface the output and ask the user whether to proceed or fix first.

### Deep checks (run in parallel if multiple selected)

**d — QA agent**

Spawn a QA subagent. Pass it: the ticket acceptance criteria (from `ticket-breakdown.md`), the absolute repo path, and the ticket branch. Ask it to verify each AC is addressed in the code. Report findings before proceeding.

**e — Security agent**

Spawn a security subagent. Pass it: the diff output and the absolute repo path. Ask it to flag any security issues (injection, auth, exposure of sensitive data, unsafe dependencies). Report findings before proceeding.

**f — Code review agent**

Spawn a code review subagent. Pass it: the changed files and the absolute repo path. Ask it to review for bugs, anti-patterns, and code quality issues. Report findings before proceeding.

After all selected checks complete, ask:

```
All checks done. Ready to push?
  [y] Push the PR
  [n] I want to make changes first
```

Wait for confirmation before pushing.

---

## Step 3 — Push and open PR

For each repo in the workspace:

**1. Push the ticket branch:**

```bash
git -C /absolute/path/to/repo push origin <TICKET_ID>
```

**2. Open a PR using `gh pr create`:**

```bash
gh pr create \
  --title "[<TICKET_ID>] <summary>" \
  --body "$(cat /Users/<username>/xnet/tickets/<TICKET_ID>/proposed-solution.md)" \
  --base <BASE_BRANCH> \
  --head <TICKET_ID>
```

**3. Output the PR URL:**

```
✅ PR opened for <repo-name>:
   <PR URL>
```

Repeat for each repo. If a repo has no changes committed, skip it and note it.

---

## Error Handling

| Situation | Action |
|-----------|--------|
| `git push` fails (auth) | Ask user to check `gh auth status` and retry |
| `git push` fails (diverged) | Show error — do not force push. Ask user to resolve manually |
| `gh` not installed | Tell user to install GitHub CLI: `brew install gh` |
| PR already exists for this branch | Output the existing PR URL, skip creation |
| Tests fail | Surface output, ask whether to proceed or fix first |
| QA/security/review agent flags issues | Surface findings, ask whether to proceed or fix first |
